1. Introduction

Part 3 modernised the Global Logistics Management System (GLMS) from a monolithic ASP.NET Core MVC application into a decoupled, containerised, service-oriented platform. The presentation layer (MVC) no longer accesses SQL Server directly; all data operations flow through a RESTful Web API secured with JWT authentication.

2. Architectural Refactoring

The Repository and Service patterns from Part 2 were preserved and relocated entirely to the API layer. The MVC application acts as a thin client, using HttpClient to consume JSON endpoints. This separation of concerns allows the frontend and backend to scale independently — for example, multiple MVC instances can share one API cluster behind a load balancer.

Key design decisions included a shared class library (GLMS.Shared) for domain models and DTOs, ensuring type consistency between API and MVC without duplicating code.

3. DevOps & Automated Testing in CI/CD

Automated testing is critical in a Continuous Integration/Continuous Deployment (CI/CD) pipeline because it provides fast, repeatable verification that code changes have not broken existing functionality. Without automated tests, developers rely on manual testing, which is slow, inconsistent, and often skipped under deadline pressure.

In a typical CI/CD workflow (e.g., GitHub Actions or Azure DevOps), every pull request triggers:

  • Unit tests — validate business logic in isolation (currency conversion, file validation, contract workflow rules).
  • Integration tests — spin up the API in-memory and call real HTTP endpoints, asserting status codes (200 OK, 201 Created, 401 Unauthorized) and verifying JSON response integrity.

Integration tests are especially valuable because they catch wiring errors — incorrect routes, missing authentication, broken serialization — that unit tests cannot detect. If a developer changes a route from /api/contracts to /api/contract, integration tests fail immediately in the pipeline, preventing the bug from reaching production.

This "shift-left" approach reduces the cost of fixing defects: a bug found in CI costs minutes to fix; the same bug in production can cost hours of downtime, customer trust, and emergency hotfixes.

4. Containerization & the "Works on My Machine" Problem

Docker solves environment inconsistency by packaging each component — database, API, and web frontend — into isolated containers with explicit dependencies, networking, and environment variables defined in docker-compose.yml.

The classic "it works on my machine" problem occurs when developers run different OS versions, SQL Server editions, or .NET SDK versions. A feature that works locally may fail on a lecturer's machine or in production. Docker eliminates this by ensuring:

  • Identical runtime — same .NET 9 ASP.NET image on every machine.
  • Identical database — SQL Server 2022 container with the same SA password and connection string.
  • Identical networking — containers communicate via Docker service names (e.g., glms-backend-api), not localhost IPs that differ per environment.
  • Reproducible builds — multi-stage Dockerfiles compile and publish in a clean environment every time.

Our docker-compose.yml orchestrates three containers: sql-server-db, glms-backend-api, and glms-frontend-web. A single command (docker compose up --build) starts the entire ecosystem, making onboarding and demonstration straightforward.

5. Security Considerations

JWT (JSON Web Token) authentication was implemented to secure API endpoints. The MVC frontend obtains a token via POST /api/auth/login and attaches it as a Bearer token on subsequent requests. This stateless authentication model scales well in distributed deployments and is industry standard for SOA and microservice architectures.

6. Conclusion

Part 3 demonstrates the evolution from prototype to production-ready architecture. Automated integration tests guard against regressions in CI/CD pipelines, while Docker ensures consistent deployment across development, testing, and production environments. Together, these practices embody modern DevOps principles and prepare GLMS for cloud-native scaling at TechMove Logistics.
